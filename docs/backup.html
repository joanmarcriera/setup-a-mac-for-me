Backup

Time Machine to TrueNAS is good enough if it already covers the job.

If Time Machine to the TrueNAS or NFS target gives a full backup and a restore path you trust, there is no need to add Kopia just for the sake of it. Keep backup simple unless a real gap shows up.

Default answer: use Time Machine. Treat Kopia as optional only when you want an extra encrypted offsite copy, repository verification, or a second backup shape beyond Time Machine.

If the goal is a full wipe, use Factory Reset as the companion page. Time Machine is the full-machine safety net before Erase Assistant.

Recommended Backup Shape

Start with the smaller plan first. Add more only if Time Machine is not covering something important.

  1. Use Time Machine with the TrueNAS or NFS destination and confirm backups complete normally.
  2. Before a reset or major upgrade, confirm the latest Time Machine snapshot exists and can be browsed.
  3. Keep app exports and small irreplaceable files in a separate private path if you want faster selective restore.
  4. Add Kopia only if you still need encrypted offsite storage or repository-style verification beyond Time Machine.

Time Machine Checks

These are the commands worth keeping close if Time Machine is the main backup path.

Latest backup

Confirm the most recent Time Machine snapshot exists.

tmutil latestbackup

Backup destination

Confirm which destination Time Machine is using.

tmutil destinationinfo

Optional Kopia extras

Only relevant if Time Machine is not enough and you want the extra layer.

brew install kopia
brew install --cask kopiaui google-drive

What To Protect

These are the paths that matter more than generic app binaries, because Homebrew can reinstall apps but it cannot recreate state.

Automation

~/Library/Application Support/Keyboard Maestro, exported macro folders, and any text expander source files.

Knowledge

Obsidian vaults, Logseq graphs, plugin settings, and any repo-backed knowledge projects.

Mail and identity

MailMate settings, dotfiles, SSH material, repo clones, and the small app preference directories that would be annoying to rebuild.

Exclude the noise

Skip caches, node_modules, derived build output, downloaded dependencies, and media you can already re-fetch.

Keyboard Maestro And GitHub

GitHub is useful here, but not as a live sync replacement.

Keep live sync simple

Continue using Google Drive for live Keyboard Maestro sync if it is already stable. That is the low-risk path.

Use git for snapshots

Export all macros as a folder and store those exports in a private Git repo for version history and rollback.

Do the same for Logseq

Logseq file graphs are a good fit for a private Git repo, which also gives you change history across notes and config.
